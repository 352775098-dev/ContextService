## 架构摘要

Context Service 是 Agent Runtime 的上下文编排与治理层，采用 **分层单体架构** 与 **依赖注入模式**，基于 **Go 1.21+** 语言实现，使用 **内存态存储 + 文件系统配置** 作为 Phase 1 基础设施。核心质量目标是 **一致性**（统一上下文读写路径）、**可控性**（明确的生命周期管理）和 **可观测性**（完整链路追踪），支持通过 **MemoryProvider** 和 **ToolProvider** 抽象层接入外部记忆服务和工具服务。

## 一、用例视图（Use Case View）

### 1.1 系统上下文模型

#### 系统定位

Context Service 是 Agent Runtime 的上下文编排与治理层（Context Orchestration Layer），负责统一调度会话与记忆上下文，并按 `agent_id + session_id` 维护会话实例、模型调用流程与步骤状态，最终将不同来源的上下文统一装配为可推理输入。

#### 系统边界

- **系统内**：
  - 会话实例与流程步骤状态管理
  - 上下文窗口信息填充与组装
  - 内核上下文版本管理与加载
  - 工具上下文参数填充与结果治理
  - 上下文治理能力（过期、回收、审计日志、降级控制）
  - 快照查询、回放与调试接口
- **系统外**：
  - KMM 知识记忆服务（Memory Provider）
  - 工具服务（Tool Provider）
  - 模型服务（LLM）
  - Agent服务

#### 核心用例 1：会话初始化

**参与者**：Agent Runtime、Context Service、File Store

**前置条件**：
- `session_id`、`agent_id`、`kernel_id`、`user_id`、`caller_id`、`trace_id` 参数有效

**主成功流程**：
1. Gateway 接收 `POST /context/v1/sessions:init` 请求
2. ContextManager 校验参数，按 `agent_id + session_id` 创建/加载会话实例
3. KernelContext 解析 `kernel_id` 并返回 prompt 参数 schema
4. ContextManager 在会话实例中登记模型调用流程与当前步骤
5. 返回会话实例元数据给上层 Agent

**异常流程**：
- 参数校验失败：返回 `ErrInvalidParameter` (400)
- 内核不存在：返回 `ErrSessionNotFound` (404)
- 会话版本冲突：返回 `ErrSessionVersionConflict` (409)

#### 核心用例 2：上下文组装

**参与者**：Agent Runtime、Context Service、KMM、Tool Provider

**前置条件**：
- 会话已初始化且状态为 `active`

**主成功流程**：

1. Gateway 接收 `POST /context/v1/sessions:assemble` 请求
2. ContextAssembler 从 ContextManager 获取当前步骤及参数定义
3. 按参数定义并行调用：
   - SessionContext（窗口计算与压缩结果替换）
   - MemoryContext -> MemoryAdapter -> KMM（长期记忆、历史会话）
   - ToolContext -> ToolAdapter -> 工具服务（如步骤需要）
4. ContextAssembler 合并所有片段并做 token 预算收敛
5. 返回 `assembled_context`、`token_usage` 与 `degraded` 信息

**异常流程**：
- 记忆服务超时：降级为仅窗口 + 内核上下文，`degraded=true`
- 工具服务超时：跳过工具参数并标记缺失
- Token 超限：按优先级裁剪，返回 `ErrTokenBudgetExceeded` (400)

#### 核心用例 4：上下文压缩

**参与者**：Agent Runtime、Context Service、KMM

**前置条件**：
- 会话已初始化且状态为 `active`
- 历史会话内容超过 Token 预算阈值

**主成功流程**：

1. ContextCompressor 接收压缩请求，指定压缩策略和参数
2. 根据压缩策略选择压缩算法：
   - **摘要压缩**：使用 LLM 生成历史对话摘要
   - **向量压缩**：使用向量检索提取关键信息
   - **规则压缩**：基于预定义规则裁剪冗余内容
   - **分层压缩**：结合摘要和向量压缩的多级策略
3. 调用 KMM 获取历史会话轮次数据
4. 执行压缩算法，生成压缩后的上下文片段
5. 计算压缩率（原始 Token 数 / 压缩后 Token 数）
6. 更新会话的压缩历史记录
7. 返回压缩结果：`compressed_context`、`compression_ratio`、`compression_method`

**异常流程**：
- 压缩服务超时：降级为简单截断策略，`degraded=true`
- 压缩率不达标（< 30%）：记录告警并返回原内容
- KMM 服务不可用：使用本地缓存的历史数据进行压缩

**压缩策略配置**：

| 策略名称 | 适用场景 | 压缩率目标 | 质量要求 |
|----------|----------|------------|----------|
| `summary_lite` | 轻量级摘要 | 40-50% | 保留关键决策点 |
| `summary_deep` | 深度摘要 | 60-70% | 完整语义保留 |
| `vector_extract` | 关键信息提取 | 50-60% | 高相关性保留 |
| `rule_truncate` | 紧急降级 | 20-30% | 保证可用性 |
| `hierarchical` | 综合策略 | 50-65% | 自适应平衡 |
| `smart_pruning` | 智能裁剪（冗余+场景感知） | 45-60% | 精准保留关键信息 |

#### 详细设计方案：融合冗余信息裁剪和场景感知的历史对话压缩

**方案名称**：SmartPruningCompression（智能裁剪压缩）

**设计目标**：
- 自动识别并裁剪工具调用、思考过程、推荐内容等低价值信息
- 基于场景感知保留对当前对话有价值的上下文
- 在保证语义完整性的前提下实现高压缩率

**核心处理流程**：


**步骤1：内容分类标记（Content Classification）**

对每轮历史对话进行多维度标记：

| 标记维度 | 类型值 | 识别特征 | 处理策略 |
|----------|--------|----------|----------|
| `message_type` | `user_query` | 用户输入 | 保留完整 |
| `message_type` | `assistant_response` | 助手回复 | 保留完整 |
| `message_type` | `tool_call` | 工具调用及结果 | 智能裁剪 |
| `message_type` | `thinking_process` | 思考过程（<think>） | 压缩或删除 |
| `message_type` | `recommendation` | 推荐内容 | 摘要保留 |
| `message_type` | `system_event` | 系统事件 | 条件保留 |
| `content_scene` | `question_answering` | 问答场景 | 保留Q&A对 |
| `content_scene` | `task_execution` | 任务执行 | 保留结果和异常 |
| `content_scene` | `multi_turn_chat` | 多轮闲聊 | 提取关键决策点 |
| `content_scene` | `tool_chaining` | 工具链调用 | 保留最终输出 |
| `value_level` | `critical` | 关键决策/承诺 | 必须保留 |
| `value_level` | `high` | 重要信息 | 优先保留 |
| `value_level` | `medium` | 一般信息 | 可摘要 |
| `value_level` | `low` | 冗余信息 | 可裁剪 |

**步骤2：冗余信息裁剪（Redundant Information Pruning）**

针对不同类型内容的裁剪策略：

```cpp
// 工具调用裁剪策略
struct ToolCallPruningStrategy {
    // 成功且无副作用的工具调用：只保留结果摘要
    bool PruneSuccessfulTool(const ToolCall& tool) {
        if (tool.status == "success" && !tool.has_side_effect) {
            return true; // 裁剪详细过程，只保留: "已执行[tool_name]，获得[结果摘要]"
        }
        return false;
    }

    // 失败或异常工具调用：保留完整信息用于错误追踪
    bool PruneFailedTool(const ToolCall& tool) {
        return tool.status != "success"; // 保留完整错误信息
    }

    // 工具链调用：只保留最终输出，中间过程用省略号代替
    std::string CompressToolChain(const std::vector<ToolCall>& chain) {
        if (chain.size() <= 2) return FormatFullChain(chain);
        return "执行了" + std::to_string(chain.size()) + "个工具，最终获得: "
               + chain.back().result_summary;
    }
};

// 思考过程裁剪策略
struct ThinkingProcessStrategy {
    // 将详细思考过程压缩为一句话决策理由
    std::string CompressThinking(const std::string& thinking) {
        // 使用轻量级模型提取核心推理路径
        return ExtractKeyDecisionPoint(thinking);
    }
};

// 推荐内容裁剪策略
struct RecommendationPruningStrategy {
    // 用户采纳的推荐：保留关键信息
    // 用户忽略的推荐：完全删除
    std::string PruneRecommendation(const Recommendation& rec, bool accepted) {
        if (!accepted) return ""; // 完全删除未采纳的推荐
        return "推荐使用: " + rec.key_point; // 只保留关键点
    }
};
```

**步骤3：场景意图识别（Scene-Aware Detection）**

基于当前对话上下文识别场景类型：

| 场景类型 | 识别特征 | 压缩策略 |
|----------|----------|----------|
| `code_debug` | 代码错误、调试请求 | 保留错误信息、保留代码片段、删除成功执行日志 |
| `data_analysis` | 数据分析、图表请求 | 保留数据源、保留结论、删除中间计算过程 |
| `creative_writing` | 创意写作、内容生成 | 保留风格要求、保留修改历史、删除生成过程中的备选 |
| `task_planning` | 任务规划、日程安排 | 保留最终计划、保留变更记录、删除讨论过程 |
| `knowledge_qa` | 知识问答 | 保留问题和准确答案、删除搜索过程 |
| `multi_agent_coord` | 多Agent协作 | 保留协调结果、保留关键决策、删除协商过程 |

**步骤4：价值分数计算（Value Scoring）**

综合多维度计算每轮对话的价值分数（0-100）：

```
价值分数 = w1 * 时效性分数 + w2 * 场景相关性 + w3 * 语义完整性 + w4 * 用户关注度

其中：
- 时效性分数：基于时间衰减函数 e^(-λ * Δt)，最近对话分数高
- 场景相关性：与当前 query 的向量相似度
- 语义完整性：是否包含完整的问答对或任务闭环
- 用户关注度：用户是否追问、是否标记重要

权重配置（可基于场景调整）：
- 问答场景: w1=0.2, w2=0.4, w3=0.3, w4=0.1
- 任务场景: w1=0.3, w2=0.3, w3=0.2, w4=0.2
```

**步骤5：语义片段合并（Semantic Chunk Merging）**

将保留下来的片段智能合并为连贯的摘要：

```cpp
// 合并策略
struct SemanticMergeStrategy {
    // 将相邻的短对话合并为一个语义单元
    std::vector<ContextChunk> MergeAdjacentChunks(
        const std::vector<ContextChunk>& chunks) {
        std::vector<ContextChunk> merged;
        for (const auto& chunk : chunks) {
            if (!merged.empty() && CanMerge(merged.back(), chunk)) {
                merged.back() = Merge(merged.back(), chunk);
            } else {
                merged.push_back(chunk);
            }
        }
        return merged;
    }

    // 判断两个片段是否可以合并
    bool CanMerge(const ContextChunk& a, const ContextChunk& b) {
        // 同一话题、时间接近、角色互补
        return SameTopic(a, b) &&
               TimeDelta(a, b) < 60s &&
               ComplementaryRole(a.role, b.role);
    }
};
```

**主成功流程**：

1. ContextSelector 接收选择请求，包含当前用户 query 和候选上下文列表
2. 根据选择模式执行不同策略：
   - **语义相关性选择**：计算候选内容与当前 query 的语义相似度
   - **时序重要性选择**：基于时间衰减模型评估历史重要性
   - **结构化选择**：按预定义结构（如技能、领域、意图）筛选
   - **混合选择**：综合语义、时序和结构化因素
3. 为每个候选上下文计算选择分数
4. 根据 Token 预算筛选 Top-K 最相关的上下文
5. 对选中内容进行排序和去重
6. 返回选择结果：`selected_contexts`、`selection_scores`、`selection_method`

**异常流程**：
- 选择服务超时：使用简单的 LRU 策略选择最近上下文
- 所有候选相关性低：扩大选择范围并标记 `low_confidence=true`
- KMM 向量服务不可用：降级为基于关键词的匹配选择

**选择算法配置**：

| 算法模式 | 核心机制 | 适用场景 | 复杂度 |
|----------|----------|----------|--------|
| `semantic_similarity` | 向量余弦相似度 | 开放式对话 | O(n) |
| `temporal_decay` | 时间衰减 + 访问频率 | 长期会话 | O(n log n) |
| `structured_match` | 元数据标签匹配 | 结构化任务 | O(log n) |
| `hybrid_scoring` | 多维度加权评分 | 复杂场景 | O(n log n) |
| `attention_based` | 注意力机制权重 | 多轮推理 | O(n²) |
| `precision_recall` | 精准匹配+召回增强 | 精确查询场景 | O(n log n) |

#### 详细设计方案：保留与当前 Query 相关且精确的历史对话选择

**方案名称**：PrecisionRecallSelector（精准召回选择器）

**设计目标**：
- 精确识别与当前 query 语义相关的历史对话
- 保留对当前决策有实质影响的关键上下文
- 避免信息过载，选择最精炼有效的历史片段

**核心处理流程**：

```mermaid
graph TD
    START([输入当前Query]) --> ANALYSIS[步骤1: Query意图分析]
    ANALYSIS --> RETRIEVAL[步骤2: 多路召回]
    RETRIEVAL --> RANKING[步骤3: 精准排序]
    RANKING --> DIVERSITY[步骤4: 多样性过滤]
    DIVERSITY --> FINAL[步骤5: 最终精选]
    FINAL --> OUTPUT[输出选中历史]

    ANALYSIS --> |提取| A1[关键词]
    ANALYSIS --> |提取| A2[意图类型]
    ANALYSIS --> |提取| A3[实体信息]
    ANALYSIS --> |提取| A4[时序约束]

    RETRIEVAL --> |召回通道| R1[语义相似]
    RETRIEVAL --> |召回通道| R2[关键词匹配]
    RETRIEVAL --> |召回通道| R3[实体关联]
    RETRIEVAL --> |召回通道| R4[时序邻近]
```

**步骤1：Query意图分析（Query Intent Analysis）**

深度解析当前 query，提取选择所需的关键要素：

```cpp
struct QueryAnalysis {
    // 语义向量表示
    std::vector<float> semantic_vector;

    // 提取的关键词及其权重
    std::vector<std::pair<std::string, float>> keywords;

    // 意图类型分类
    enum IntentType {
        FOLLOW_UP,      // 追问/跟进
        TOPIC_SWITCH,   // 话题切换
        REFINEMENT,     // 细化/澄清
        CONTRADICTION,  // 否定/修正
        GENERAL         // 一般询问
    } intent_type;

    // 命名实体识别结果
    struct Entity {
        std::string name;
        std::string type;  // PERSON, LOCATION, ORGANIZATION, TIME, etc.
        float confidence;
    };
    std::vector<Entity> entities;

    // 时序约束（如"刚才说的","之前提到的"）
    struct TemporalConstraint {
        bool has_constraint;
        enum Type { RECENT, SPECIFIC, RANGE } type;
        int turn_range;  // 最近N轮
    } temporal;

    // 话题一致性检测
    float topic_coherence_with_recent;  // 与最近对话的话题连贯度 0-1
};
```

**步骤2：多路召回（Multi-Channel Retrieval）**

基于不同策略召回候选历史对话：

| 召回通道 | 策略描述 | 适用场景 | 召回数量 |
|----------|----------|----------|----------|
| **语义相似召回** | 向量相似度 Top-K | 语义相关但表述不同 | K=10 |
| **关键词匹配召回** | BM25/TF-IDF 高分 | 关键词重叠 | K=8 |
| **实体关联召回** | 相同实体共现 | 涉及相同人物/地点/事物 | K=6 |
| **时序邻近召回** | 最近 N 轮对话 | 上下文连贯性强 | K=5 |
| **意图链召回** | 相同意图类型的历史 | 追问、细化类 query | K=5 |
| **用户关注召回** | 用户标记重要/追问 | 高价值历史 | K=3 |

**步骤3：精准排序（Precision Ranking）**

使用多阶段排序模型对召回结果进行精细排序：

```cpp
// 第一阶段：粗排（轻量级特征，快速过滤）
struct CoarseRanker {
    float Score(const HistoryTurn& turn, const QueryAnalysis& query) {
        float score = 0.0;

        // 1. 基础语义相似度 (40%)
        score += 0.4 * CosineSimilarity(turn.vector, query.semantic_vector);

        // 2. 关键词匹配度 (30%)
        score += 0.3 * KeywordMatchScore(turn.content, query.keywords);

        // 3. 实体匹配度 (20%)
        score += 0.2 * EntityOverlapScore(turn.entities, query.entities);

        // 4. 时序衰减 (10%)
        score += 0.1 * TemporalDecayScore(turn.timestamp);

        return score;
    }
};

// 第二阶段：精排（深度特征，保证质量）
struct FineRanker {
    float Score(const HistoryTurn& turn, const QueryAnalysis& query) {
        float score = 0.0;

        // 1. 语义相关性（使用交叉编码器）
        score += 0.35 * CrossEncoderScore(query.text, turn.content);

        // 2. 意图一致性
        score += 0.25 * IntentConsistencyScore(query.intent_type, turn.intent);

        // 3. 话题连贯性
        score += 0.20 * TopicCoherenceScore(query.text, turn.topic_vector);

        // 4. 信息增益评估（该历史对回答当前query的价值）
        score += 0.15 * InformationGainScore(turn, query);

        // 5. 用户反馈信号（历史是否被用户确认/采纳）
        score += 0.05 * UserFeedbackScore(turn.feedback_signals);

        return score;
    }
};
```

**步骤4：多样性过滤（Diversity Filtering）**

避免选择内容高度相似的历史，保证信息的多样性：

```cpp
// MMR (Maximal Marginal Relevance) 算法实现
std::vector<HistoryTurn> DiversityFilter(
    const std::vector<HistoryTurn>& ranked_history,
    const QueryAnalysis& query,
    float lambda = 0.5) {  // 平衡相关性和多样性

    std::vector<HistoryTurn> selected;
    std::set<size_t> selected_indices;

    while (selected.size() < target_count && selected_indices.size() < ranked_history.size()) {
        float max_mmr_score = -1;
        size_t best_idx = 0;

        for (size_t i = 0; i < ranked_history.size(); ++i) {
            if (selected_indices.count(i)) continue;

            // MMR = λ * Sim(query, doc) - (1-λ) * max(Sim(doc, selected))
            float relevance = Similarity(query, ranked_history[i]);
            float max_sim_to_selected = 0;
            for (const auto& sel : selected) {
                max_sim_to_selected = std::max(max_sim_to_selected,
                                               Similarity(ranked_history[i], sel));
            }

            float mmr_score = lambda * relevance - (1 - lambda) * max_sim_to_selected;

            if (mmr_score > max_mmr_score) {
                max_mmr_score = mmr_score;
                best_idx = i;
            }
        }

        selected.push_back(ranked_history[best_idx]);
        selected_indices.insert(best_idx);
    }

    return selected;
}
```

**步骤5：最终精选（Final Selection）**

基于 Token 预算和重要性进行最终筛选：

```
精选策略：
1. 强制保留（Must-Have）：
   - 包含与当前 query 相同实体的对话
   - 用户明确确认/采纳的历史建议
   - 上一轮的直接上下文（保持连贯性）

2. 优先保留（Should-Have）：
   - 高排序分数且信息增益大
   - 与当前话题高度相关
   - 包含关键决策点或用户偏好

3. 可选保留（Nice-to-Have）：
   - 提供额外背景信息
   - 有助于理解用户长期偏好

4. 排除（Exclude）：
   - 与当前 query 语义无关
   - 已被后续对话覆盖/修正
   - 信息价值低（如简单问候）
```

**特殊场景处理**：

| 场景 | 检测特征 | 选择策略 |
|------|----------|----------|
| **追问场景** | "刚才说的..."、"那个..." | 优先选择最近的包含目标实体的对话 |
| **话题回归** | "回到之前的话题..." | 选择该话题的历史锚点对话 |
| **修正/否定** | "不对，我是说..." | 选择被修正前的对话（了解误解原因）和修正后的对话 |
| **多轮规划** | "第一步...第二步..." | 选择规划相关的完整对话链 |
| **对比询问** | "A和B有什么区别..." | 分别选择包含A和B的历史对话 |

**相关性评估示例**：

```
当前Query: "那个方案的成本是多少？"

历史候选：
1. [5轮前] 用户: 帮我设计一个微服务架构方案
   → 相关性: 0.85 (高，是提到的"那个方案")

2. [3轮前] 用户: 数据库选型有什么建议？
   → 相关性: 0.45 (中，属于架构的一部分)

3. [2轮前] 用户: 推荐一本技术书
   → 相关性: 0.10 (低，无关话题)

4. [1轮前] 用户: 这个方案需要多少人天？
   → 相关性: 0.90 (高，同一方案的成本相关)

选择结果: [1, 4] + 当前query构成的完整上下文
```

**配置参数**：

| 参数名 | 默认值 | 说明 |
|--------|--------|------|
| `recall_channels` | 6 | 启用的召回通道数量 |
| `coarse_top_k` | 20 | 粗排保留数量 |
| `fine_top_k` | 10 | 精排保留数量 |
| `diversity_lambda` | 0.5 | MMR多样性权重 |
| `max_token_budget` | 2000 | 选择结果的最大Token数 |
| `min_relevance_threshold` | 0.3 | 最小相关性阈值 |
| `preserve_last_n` | 2 | 强制保留最近N轮 |
| `enable_entity_boost` | true | 启用实体匹配加分 |

**效果评估指标**：

| 指标 | 说明 | 目标值 |
|------|------|--------|
| 精确率@K | 前K个选择结果的相关比例 | ≥ 90% |
| 召回率 | 相关历史被成功选择的比例 | ≥ 85% |
| 多样性指数 | 选择结果的多样性度量 | ≥ 0.6 |
| 用户满意度 | 人工评估选择质量 | ≥ 4.2/5 |

#### 核心用例 6：Local Context Cache 上下文获取

**参与者**：业务 Agent、Local Context Cache、Context Service、LLM

**前置条件**：
- 业务 Agent 已创建 AgentContext 实例
- Workflow 流程已定义并传入
- Context Service 服务可用

**主成功流程**：

1. 业务 Agent 创建 AgentContext 对象，传入 workflow 配置
2. AgentContext 解析 workflow，识别各节点所需的上下文类型
3. AgentContext 启动异步预加载线程，提前获取可能需要的上下文
4. Agent 执行到某 node 节点时，调用 AgentContext.GetContext(node_id)
5. AgentContext 检查缓存：命中直接返回，未命中则同步请求 Context Service
6. AgentContext 组装完整的上下文信息（包括 prompt、memory、tool 等）
7. 返回 ContextPackage 给业务 Agent
8. 业务 Agent 将上下文传给 LLM 进行推理

**异常流程**：
- Context Service 超时：返回已缓存的基础上下文，标记 `partial=true`
- 上下文获取失败：返回错误，业务 Agent 可选择降级或重试
- Workflow 节点不存在：返回 `ErrInvalidNodeID` 错误

#### 核心用例 7：缓存一致性管理

**参与者**：Local Context Cache、Context Service、Agent Runtime

**前置条件**：
- Local Context Cache 已缓存部分上下文数据
- Context Service 中对应的数据可能已更新

**主成功流程**：

1. Local Context Cache 定期或在特定事件触发时，向 Context Service 发送版本校验请求
2. Context Service 返回各缓存键的当前版本号（Version Vector）或 ETag
3. Local Cache 对比本地版本与服务器版本
4. 若版本不一致，标记该缓存项为 `stale`
5. 根据策略选择立即刷新或惰性刷新（下次访问时更新）
6. 更新本地缓存版本号，保持与服务端一致

**异常流程**：
- 版本校验超时：标记缓存为 `uncertain`，增加校验频率
- 服务端不可达：延长本地缓存有效期，进入降级模式
- 版本冲突（Concurrent Update）：使用 Last-Write-Wins 或自定义冲突解决策略

| 失效级别 | 触发条件 | 处理策略 | 性能影响 |
|----------|----------|----------|----------|
| **Level 1: 软失效** | 版本号变化但内容未变（哈希相同） | 仅更新版本号，保留缓存内容 | 无 |
| **Level 2: 内容失效** | 服务端内容更新 | 标记为 stale，下次访问时刷新 | 低 |

```cpp
// 内部实现使用 Pimpl 模式隐藏线程安全细节
class AgentContext::Impl {
public:
    // 缓存访问控制
    struct ThreadSafeCache {
        mutable std::shared_mutex cache_mutex;  // C++17 读写锁
        std::unordered_map<std::string, CacheEntry> data;

        // 读操作（共享锁）
        std::optional<CacheEntry> Get(const std::string& key) const {
            std::shared_lock<std::shared_mutex> lock(cache_mutex);
            auto it = data.find(key);
            if (it != data.end()) return it->second;
            return std::nullopt;
        }

        // 写操作（独占锁）
        void Set(const std::string& key, const CacheEntry& value) {
            std::unique_lock<std::shared_mutex> lock(cache_mutex);
            data[key] = value;
        }

        // 批量读（共享锁）
        std::vector<CacheEntry> GetBatch(const std::vector<std::string>& keys) const {
            std::shared_lock<std::shared_mutex> lock(cache_mutex);
            std::vector<CacheEntry> results;
            for (const auto& key : keys) {
                auto it = data.find(key);
                if (it != data.end()) results.push_back(it->second);
            }
            return results;
        }
    };

    // 任务队列（预加载用）
    struct PreloadTaskQueue {
        std::mutex queue_mutex;
        std::condition_variable cv;
        std::queue<PreloadTask> tasks;
        std::atomic<size_t> pending_count{0};
        std::atomic<bool> shutdown{false};

        void Push(PreloadTask task) {
            std::lock_guard<std::mutex> lock(queue_mutex);
            tasks.push(std::move(task));
            pending_count++;
            cv.notify_one();
        }

        std::optional<PreloadTask> Pop() {
            std::unique_lock<std::mutex> lock(queue_mutex);
            cv.wait(lock, [this] { return !tasks.empty() || shutdown; });
            if (shutdown && tasks.empty()) return std::nullopt;
            auto task = std::move(tasks.front());
            tasks.pop();
            pending_count--;
            return task;
        }
    };

    // 会话状态（需原子操作）
    struct AtomicSessionState {
        std::atomic<std::string*> current_task_id{nullptr};
        std::atomic<uint64_t> sequence_number{0};
        std::atomic<bool> is_initialized{false};
    };
};
```

```cpp
// 线程安全的数据结构封装
template<typename T>
class ThreadSafe {
public:
    template<typename... Args>
    ThreadSafe(Args&&... args) : data_(std::forward<Args>(args)...) {}

    // 读访问（const）
    template<typename Func>
    auto Read(Func&& func) const -> decltype(func(std::declval<const T&>())) {
        std::shared_lock<std::shared_mutex> lock(mutex_);
        return func(data_);
    }

    // 写访问（非const）
    template<typename Func>
    auto Write(Func&& func) -> decltype(func(std::declval<T&>())) {
        std::unique_lock<std::shared_mutex> lock(mutex_);
        return func(data_);
    }

private:
    mutable std::shared_mutex mutex_;
    T data_;
};

// 使用示例

// 读操作

    auto it = cache.find("node_1");
    return it != cache.end() ? std::optional(it->second) : std::nullopt;
});

// 写操作
```

## 二、逻辑视图（Logical View）

### 2.1 结构模型

#### 架构风格选型

**选定架构风格**：分层单体架构（Layered Monolith）+ 依赖注入（DI）模式

**理由**：
- Phase 1 范围聚焦于核心上下文编排能力，团队规模小，单体架构更合适
- 分层清晰（Gateway -> Manager -> Assembler -> Context -> Adapter），便于后续拆分微服务
- 依赖注入模式支持南向依赖可插拔（MemoryProvider、ToolProvider、CacheProvider）

**备选方案对比**：

| 方案 | 优点 | 缺点 | Trade-off |
|------|------|------|----------|
| 分层单体 | 开发简单、部署便捷、事务一致性好 | 后期拆分成本 | Phase 1 优先选择 |
| 微服务 | 独立扩缩容、技术栈灵活 | 运维复杂、分布式事务难 | 不符合 Phase 1 约束 |
| 事件驱动 | 解耦强、异步高吞吐 | 一致性保证复杂、调试困难 | 不符合 Phase 1 范围 |

### 2.2 行为模型

#### 关键业务流程 1：会话生命周期

```mermaid
stateDiagram-v2
    [*] --> ACTIVE: 创建会话（InitSession）
    ACTIVE --> CLOSED: 主动关闭
    ACTIVE --> EXPIRED: 超时（TTL到期）
    EXPIRED --> [*]
    CLOSED --> [*]

    note right of ACTIVE
        会话可执行以下操作：
        - 上下文组装
        - 快照查询
        - 步骤状态更新
    end note
```

#### 关键业务流程 2：步骤状态流转

```mermaid
stateDiagram-v2
    [*] --> PENDING: 创建步骤
    PENDING --> RUNNING: 开始组装
    RUNNING --> COMPLETED: 组装成功
    RUNNING --> FAILED: 组装失败
    FAILED --> PENDING: 重试
    COMPLETED --> [*]
    FAILED --> [*]

    note right of RUNNING
        触发降级条件：
        - 记忆服务超时
        - 工具服务不可用
        - Token 预算超限
    end note
```

#### 关键业务流程 3：上下文组装流程

```mermaid
graph TD
    START([开始组装]) --> PARSE[解析当前步骤参数计划]
    PARSE --> PARALLEL{并行获取上下文}

    PARALLEL -->|窗口| SESSION[SessionContext<br/>构建会话窗口]
    PARALLEL -->|记忆| MEMORY[MemoryContext<br/>检索长期记忆]
    PARALLEL -->|内核| KERNEL[KernelContext<br/>加载内核配置]
    PARALLEL -->|工具| TOOL[ToolContext<br/>准备工具参数]

    SESSION --> MERGE[合并上下文片段]
    MEMORY --> MERGE
    KERNEL --> MERGE
    TOOL --> MERGE

    MERGE --> TOKEN[Token 预算裁剪]
    TOKEN --> CHECK{Token 超限?}

    CHECK -->|否| UPDATE[更新步骤状态]
    CHECK -->|是| TRIM[按优先级裁剪]
    TRIM --> UPDATE

    UPDATE --> RETURN[返回组装结果]
    RETURN --> END([结束])

    alt 南向依赖失败
        MEMORY -.->|超时| DEGRADE[降级处理]
        TOOL -.->|超时| DEGRADE
        DEGRADE --> SKIP[跳过失败组件]
        SKIP --> MERGE
    end
```

#### 事务边界与一致性

- **强一致性**：会话状态更新使用乐观并发控制（`session_version`）
- **最终一致性**：南向依赖调用失败时，降级返回最小可用上下文，不阻塞主链路
- **幂等性**：`assemble` 接口通过 `session_version` 防止重复组装

## 六、架构决策记录（ADR）

### ADR-001：采用分层单体架构

- **日期**：2026-03-01
- **状态**：已接受
- **背景**：Context Service 是 Agent Runtime 的上下文编排层，Phase 1 范围聚焦核心能力，团队规模小，需要快速交付。
- **决策**：采用分层单体架构（Layered Monolith）+ 依赖注入（DI）模式。
- **备选方案**：
  - **方案 A（分层单体）**：
    - 优点：开发简单、部署便捷、事务一致性好、易于调试
    - 缺点：后续拆分微服务需要额外成本
  - **方案 B（微服务）**：
    - 优点：独立扩缩容、技术栈灵活
    - 缺点：运维复杂、分布式事务难、开发效率低
- **后果**：
  - ✅ 正面影响：Phase 1 快速交付、部署简单、事务一致性好
  - ⚠️ 负面影响/风险：后续拆分微服务需要重构成本
  - 📌 需关注：模块边界必须清晰（Gateway -> Manager -> Assembler -> Context -> Adapter），便于后续拆分

---

### ADR-002：使用 Go 标准库而非第三方框架

- **日期**：2026-03-01
- **状态**：已接受
- **背景**：Context Service 需要处理 HTTP 请求、JSON 序列化、日志等基础能力。
- **决策**：使用 Go 标准库（net/http、encoding/json、errors、testing），不引入 Gin/Echo 等第三方 Web 框架。
- **备选方案**：
  - **方案 A（标准库）**：
    - 优点：零额外依赖、学习成本低、长期稳定
    - 缺点：缺少部分高级特性（如路由、中间件）
  - **方案 B（Gin/Echo）**：
    - 优点：路由简洁、中间件丰富、生态好
    - 缺点：引入依赖、可能升级问题
- **后果**：
  - ✅ 正面影响：零额外依赖、维护成本低、长期稳定
  - ⚠️ 负面影响/风险：需要自行实现部分特性（如路由、限流中间件）
  - 📌 需关注：中间件设计需预留扩展接口

---

### ADR-003：南向依赖通过适配层抽象

- **日期**：2026-03-01
- **状态**：已接受
- **背景**：Context Service 需要调用 KMM 记忆服务和工具服务，这些服务的协议和实现可能变化。
- **决策**：通过 MemoryProvider 和 ToolProvider 接口抽象南向依赖，所有调用通过适配层进行统一治理（超时、重试、熔断、错误映射）。
- **备选方案**：
  - **方案 A（适配层抽象）**：
    - 优点：解耦、可插拔、统一治理
    - 缺点：增加一层抽象，开发成本略高
  - **方案 B（直接调用）**：
    - 优点：开发简单
    - 缺点：耦合严重、难以切换实现、无法统一治理
- **后果**：
  - ✅ 正面影响：南向依赖可插拔、统一超时重试策略、便于测试（Mock）
  - ⚠️ 负面影响/风险：增加抽象层，初期开发成本略高
  - 📌 需关注：适配层接口需稳定，避免频繁变更

---

### ADR-004：Phase 1 使用内存存储

- **日期**：2026-03-01
- **状态**：已接受
- **背景**：Context Service 需要存储会话实例、状态等信息。Phase 1 聚焦核心能力，需要快速交付。
- **决策**：Phase 1 使用 `sync.Map` 或 `sync.RWMutex` + 内存存储，不引入 Redis/PostgreSQL 等外部依赖。
- **备选方案**：
  - **方案 A（内存存储）**：
    - 优点：零外部依赖、性能最高、快速交付
    - 缺点：重启数据丢失、无法横向扩展
  - **方案 B（Redis）**：
    - 优点：数据持久化、支持横向扩展
    - 缺点：引入外部依赖、部署复杂
  - **方案 C（PostgreSQL）**：
    - 优点：强一致性、查询能力强大
    - 缺点：引入外部依赖、ORM 层复杂
- **后果**：
  - ✅ 正面影响：零外部依赖、快速交付、性能最高
  - ⚠️ 负面影响/风险：重启数据丢失、无法横向扩展
  - 📌 需关注：预留 CacheAdapter 接口，Phase 2 可平滑切换到 Redis

### ADR-005：降级优先于阻塞主链路

- **日期**：2026-03-01
- **状态**：已接受
- **背景**：南向依赖（KMM、工具服务）可能不可用，需要确保主链路不中断。
- **决策**：南向依赖失败时，默认允许降级为"仅会话+内核上下文"路径，返回 `degraded=true` 和降级原因，主链路不中断。
- **备选方案**：
  - **方案 A（降级优先）**：
    - 优点：可用性高、主链路不中断
    - 缺点：返回内容不完整
  - **方案 B（阻塞主链路）**：
    - 优点：保证内容完整
    - 缺点：可用性低、南向故障影响全局
- **后果**：
  - ✅ 正面影响：南向故障不阻塞主链路、可用性高
  - ⚠️ 负面影响/风险：返回内容不完整，可能影响模型质量
  - 📌 需关注：降级响应必须明确标记 `degraded=true` 和 `degrade_reason`，便于上层决策

  - API Key

#### 授权模型

- **Phase 1**：无授权控制
- **Phase 2**：计划引入 RBAC（基于角色的访问控制）

#### 数据加密

- **传输加密**：Phase 1 暂不强制 TLS，生产环境建议使用 HTTPS
- **静态加密**：Phase 1 不涉及敏感数据存储

#### 敏感数据处理

- **Phase 1**：不处理 PII 数据
- **Phase 2**：需要时引入数据脱敏策略

#### 安全扫描

- **代码检查**：`go vet`, `golint`
- **依赖扫描**：`go mod verify`

### 7.2 错误处理规范

#### 错误码体系

| 类别 | 错误码 | HTTP 状态 | 说明 |
|------|--------|----------|------|
| 领域错误 | `ErrSessionNotFound` | 404 | 会话不存在或已过期 |
| 领域错误 | `ErrFlowStepInvalid` | 400 | 流程步骤无效 |
| 领域错误 | `ErrTokenBudgetExceeded` | 400 | Token 预算超限 |
| 领域错误 | `ErrAssembledContextNotFound` | 404 | 会话存在但无可读上下文内容 |
| 领域错误 | `ErrSessionVersionConflict` | 409 | 会话版本冲突 |
| 记忆依赖错误 | `ErrMemoryDependencyTimeout` | 504 | 记忆服务调用超时 |
| 记忆依赖错误 | `ErrMemoryDependencyUnavailable` | 503 | 记忆服务不可用 |
| 工具依赖错误 | `ErrToolDependencyTimeout` | 504 | 工具服务调用超时 |
| 工具依赖错误 | `ErrToolDependencyUnavailable` | 503 | 工具服务不可用 |
| 通用错误 | `ErrInvalidParameter` | 400 | 请求参数无效 |
| 通用错误 | `ErrInternalError` | 500 | 内部错误 |
| 压缩错误 | `ErrCompressionFailed` | 500 | 上下文压缩失败 |
| 压缩错误 | `ErrCompressionTimeout` | 504 | 压缩服务超时 |
| 压缩错误 | `ErrCompressionQualityLow` | 400 | 压缩质量不达标 |
| 选择错误 | `ErrSelectionFailed` | 500 | 上下文选择失败 |
| 选择错误 | `ErrSelectionTimeout` | 504 | 选择服务超时 |
| 选择错误 | `ErrNoRelevantContext` | 404 | 未找到相关上下文 |

#### 重试策略

| 场景 | 重试次数 | 退避策略 |
|------|----------|----------|
| 记忆服务超时 | 3 | 指数退避（1s, 2s, 4s） |
| 工具服务超时 | 2 | 固定延迟（1s） |
| 其他可重试错误 | 1 | 固定延迟（500ms） |

#### 幂等设计

- **Phase 1**：通过 `session_version` 实现乐观并发控制
- **Phase 2**：计划引入幂等键（`idempotency_key`）

#### 错误传播规则

- **边界处理**：Gateway 层统一错误映射，不暴露南向错误细节
- **错误链**：使用 `errors.Wrap` 或 `%w` 保留错误上下文

### 7.3 API 设计规范

#### 版本策略

- **URL 版本**：所有 API 使用 `/context/v1/` 前缀
- **版本演进**：向后兼容优先，破坏性变更需要升级版本号

#### 分页规范

- **Phase 1**：暂不支持分页
- **Phase 2**：计划使用 cursor-based 分页

#### 幂等性保障

- **Phase 1**：通过 `session_version` 实现乐观并发控制
- **Phase 2**：计划引入幂等键

### A. 术语表

| 术语 | 说明 |
|------|------|
| **Session Context** | 单次 Session 内的运行时动态状态，包含对话历史、会话窗口等实时数据 |
| **Memory Context** | 跨 Session 的长期记忆与用户画像，包括长期记忆、历史会话压缩结果等 |
| **Kernel Context** | Agent 能力定义与执行策略，包含 Prompt、Skills、MCP 与策略配置 |
| **Tool Context** | 工具参数与工具结果的上下文化表示 |
| **Context Manager** | 会话实例与流程步骤状态管理组件 |
| **Context Assembler** | 多源上下文组装与裁剪执行组件 |
| **Context Compressor** | 上下文压缩组件，负责减少历史会话的 Token 占用 |
| **Context Selector** | 上下文选择组件，负责从候选内容中选择最相关的上下文 |
| **Flow Step** | 模型调用流程中的单个步骤 |
| **Session Instance** | 由 `agent_id + session_id` 唯一标识的会话实例 |
| **Kernel Snapshot** | 内核的版本化快照 |
| **Local Context Cache** | 业务 Agent 侧本地缓存 |
| **北向接口** | 对上层 Agent 应用暴露的服务接口 |
| **南向依赖** | 对记忆服务、工具服务等基础能力的调用 |
| **Token Budget** | 模型调用可用的 token 预算 |
| **Degrade** | 当南向依赖不可用时，降级返回最小可用上下文 |
| **Window Strategy** | 会话窗口管理策略 |
| **Trace ID** | 链路追踪标识符 |
| **Compression Strategy** | 上下文压缩策略，如摘要压缩、向量压缩、规则压缩等 |
| **Compression Ratio** | 压缩率，表示压缩后 Token 数与原始 Token 数的比例 |
| **Selection Algorithm** | 上下文选择算法，如语义相似度、时序衰减、混合评分等 |
| **Context Candidate** | 上下文候选，供选择器评估和筛选的上下文片段 |
| **Relevance Score** | 相关性分数，表示候选上下文与当前查询的相关程度 |
| **AgentContext** | C++ SDK 核心类，业务 Agent 通过该类与 Context Service 交互 |
| **Local Context Cache** | 本地上下文缓存，位于业务 Agent 侧的 C++ 客户端 SDK |
| **Workflow** | 工作流定义，描述 Agent 执行流程及各节点所需上下文 |
| **ContextPackage** | 上下文包，包含完整的模型输入上下文信息 |
| **Preload** | 预加载机制，提前异步获取后续节点需要的上下文 |
| **Task** | 子任务，在 Session 内创建的独立上下文命名空间 |
